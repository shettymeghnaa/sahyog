\section{Introduction}

\subsection{Purpose}
This Software Requirements Specification (SRS) document provides a comprehensive description of the "Sahyog" Mutual Insurance platform. It captures both the functional and non-functional requirements essential for the design, development, and testing of the system. The primary goal is to ensure that all stakeholders—including developers, QA teams, project managers, and the client—have a unified understanding of the system's capabilities and constraints.

\subsection{Scope}
Project Sahyog is an advanced mutual insurance solution tailored for regional disaster resilience. Unlike traditional insurance models, Sahyog leverages a community-pooling mechanism where members of a specific region contribute to a shared fund. The system automates the lifecycle of insurance management, including:
\begin{itemize}
    \item \textbf{Onboarding:} Secular registration of members and verification.
    \item \textbf{Financial Management:} Tracking monthly contributions and maintaining pool fund solvency.
    \item \textbf{Disaster Management:} Monitoring regional disaster events and defining eligibility windows.
    \item \textbf{Claim Processing:} Transparent submission, multi-step verification by officers, and final administrative approval.
    \item \textbf{Governance:} Enforcement of policies regarding waiting periods, payout caps, and reserve percentages.
\end{itemize}

\subsection{Definitions, Acronyms, and Abbreviations}
\begin{itemize}
    \item \textbf{Member:} An individual who has registered and pays regular contributions to gain insurance coverage.
    \item \textbf{Claim Officer:} A specialized user role responsible for reviewing insurance claims filed by members.
    \item \textbf{Pool Fund:} A collective bank of funds contributed by members within a specific region.
    \item \textbf{Waiting Period:} The duration a member must wait after joining before being eligible to file a claim.
    \item \textbf{Reserve Percentage:} The minimum percentage of the pool fund that must be kept as a buffer and cannot be used for payouts.
    \item \textbf{REST:} Representational State Transfer, the architectural style for web services.
    \item \textbf{JWT:} JSON Web Token, used for secure transmission of information between parties as a JSON object.
\end{itemize}

\section{Overall Description}

\subsection{Product Perspective}
Sahyog is a self-contained insurance management suite designed to replace manual, error-prone community fund tracking. It is built as a highly scalable web application using Java for the backend (Spring Boot) and TypeScript for the frontend (Angular). The system interacts with a relational database to maintain strict ACID compliance for financial transactions.

\subsection{Product Functions}
The system is divided into five core functional modules:
\begin{enumerate}
    \item \textbf{Identity and Access Management (IAM):} Handles multi-tenant authentication, role-based authorization, and user profile management.
    \item \textbf{Regional Administration:} Allows administrators to define geographical regions, assign pool funds, and set local policies.
    \item \textbf{Contribution Ledger:} Automates the collection and tracking of monthly member contributions.
    \item \textbf{Disaster Monitoring:} Records active disaster events (e.g., floods, earthquakes) to trigger claim eligibility.
    \item \textbf{Claims Engine:} A state-machine based engine that moves claims through stages: Submitted $\rightarrow$ Under Review $\rightarrow$ Approved/Rejected $\rightarrow$ Paid.
\end{enumerate}

\subsection{User Classes and Characteristics}

\subsubsection{Member}
Members are the end-users of the insurance product. They represent individuals seeking financial protection against disasters. They primarily interact with the dashboard to view their contribution status and file claims during emergencies.
\begin{itemize}
    \item \textbf{Technical Literacy:} Low to Moderate.
    \item \textbf{Frequency of Use:} Occasional (monthly for payments, emergency for claims).
\end{itemize}

\subsubsection{Claim Officer}
Claim Officers are professionals or community leads tasked with verifying the validity of claims. They operate at the regional level and must have a high level of integrity and attention to detail.
\begin{itemize}
    \item \textbf{Technical Literacy:} Moderate.
    \item \textbf{Frequency of Use:} High (daily task review).
\end{itemize}

\subsubsection{Administrator}
Administrators are the system controllers. They manage the overall platform health, handle financial payouts, and configure the rules that govern the mutual insurance logic.
\begin{itemize}
    \item \textbf{Technical Literacy:} High.
    \item \textbf{Frequency of Use:} Routine.
\end{itemize}

\subsubsection{Government/Auditor}
Government roles provide oversight. They have read-only access to dashboards and can generate reports on fund utilization and disaster impact.
\begin{itemize}
    \item \textbf{Technical Literacy:} Moderate.
    \item \textbf{Frequency of Use:} Analytical/Periodic.
\end{itemize}

\subsection{Operating Environment}
\begin{itemize}
    \item \textbf{Hardware:} Minimum 4GB RAM, Quad-core processor for server-side hosting.
    \item \textbf{Cloud Provider:} AWS, Azure, or Google Cloud Platform.
    \item \textbf{Software Stack:}
    \begin{itemize}
        \item Backend: Java 17+, Maven 3.8+
        \item Frontend: Angular 21+, Node.js 20+
        \item Database: MySQL 8.0 / PostgreSQL 15+
    \end{itemize}
\end{itemize}

\subsection{Design and Implementation Constraints}
\begin{itemize}
    \item \textbf{Regulatory Compliance:} The system must adhere to regional data privacy laws (e.g., GDPR).
    \item \textbf{Security:} All financial transactions must be logged for audit trails.
    \item \textbf{Concurrency:} The system must handle simultaneous claim submissions from hundreds of users without data corruption.
\end{itemize}

\section{System Features and Functional Requirements}

\subsection{Module 1: Authentication and Profile Management}

\subsubsection{Description}
This module handles all aspects of user identity. It ensures that only authorized users can access specific features based on their roles.

\subsubsection{Functional Requirements}
\begin{itemize}
    \item \textbf{FR-1.1: Registration:} The system shall allow new users to register as members by providing full name, email, password, and selecting their region.
    \item \textbf{FR-1.2: Login:} The system shall authenticate users using email and encrypted passwords, returning a JWT token upon success.
    \item \textbf{FR-1.3: Role Management:} Each user shall be assigned one of the roles: MEMBER, OFFICER, ADMIN, GOVERNMENT.
    \item \textbf{FR-1.4: Profile Update:} Members shall be able to update their contact information and view their join date.
\end{itemize}

\subsection{Module 2: Member Dashboard and Contributions}

\subsubsection{Description}
This module empowers members to manage their relationship with the mutual fund.

\subsubsection{Functional Requirements}
\begin{itemize}
    \item \textbf{FR-2.1: Payment Gateway Integration:} The system shall allow members to pay their monthly contributions.
    \item \textbf{FR-2.2: Contribution Tracking:} Members shall have a history view of all past payments.
    \item \textbf{FR-2.3: Eligibility status:} The system shall display the member's current status (ACTIVE, INACTIVE, WAITING\_PERIOD).
\end{itemize}

\subsection{Module 3: Claims Management Engine}

\subsubsection{Description}
This is the core business engine of Sahyog, managing the lifecycle of an insurance claim.

\subsubsection{Functional Requirements}
\begin{itemize}
    \item \textbf{FR-3.1: Claim Submission:} Members shall be able to submit a claim by selecting an active disaster and providing a requested amount and supporting documentation.
    \item \textbf{FR-3.2: Automated Assignment:} New claims shall be automatically assigned to the Claim Officer with the lowest current workload in that region.
    \item \textbf{FR-3.3: Document Verification:} Claim Officers shall be able to download/view documents and add verification notes.
    \item \textbf{FR-3.4: Payout Calculation:} The system shall calculate the approved payout based on the policy (e.g., min(Requested, MaxPayout)).
\end{itemize}

\section{Use Case Analysis}

\subsection{Use Case UC-1: Filing an Insurance Claim}
\textbf{Primary Actor:} Member \\
\textbf{Pre-conditions:} Member is active, contribution for current month is paid, and an active disaster is declared in their region. \\
\textbf{Main Success Scenario:}
\begin{enumerate}
    \item Member logs into the Sahyog portal.
    \item Member navigates to "Submit Claim".
    \item Member selects the relevant Disaster Event.
    \item Member enters the requested amount and uploads proof (e.g., photos).
    \item System validates eligibility (Waiting period, Region match).
    \item System saves the claim and assigns it to an officer.
    \item Member receives a confirmation notification.
\end{enumerate}

\subsection{Use Case UC-2: Reviewing a Claim}
\textbf{Primary Actor:} Claim Officer \\
\textbf{Pre-conditions:} A claim has been assigned to the officer. \\
\textbf{Main Success Scenario:}
\begin{enumerate}
    \item Officer logs in and views their dashboard.
    \item Officer selects a "Submitted" claim.
    \item Officer reviews the description and uploaded documents.
    \item Officer changes status to "Under Review".
    \item Officer provides a final decision (Approve/Reject) with notes.
    \item System updates the claim status and notifies the Admin.
\end{enumerate}

\section{Data Requirements}

\subsection{Logical Data Model}
The following entities represent the core data structure:
\begin{itemize}
    \item \textbf{Member:} ID, Name, Email, RegionID, TotalContribution, JoinDate.
    \item \textbf{Region:} ID, Name, TotalMembers, TotalFund.
    \item \textbf{Policy:} RegionID, MonthlyContribution, WaitingPeriod, MaxPayout.
    \item \textbf{Claim:} ID, MemberID, DisasterID, Amount, Status, AssignedOfficerID.
    \item \textbf{PoolFund:} RegionID, Balance, ReservePercentage.
\end{itemize}

\section{Quality Attributes (Non-Functional Requirements)}

\subsection{Reliability}
The system shall maintain 99.9\% availability during business hours. Financial data must be backed up daily.

\subsection{Security}
\begin{itemize}
    \item All passwords must be hashed using BCrypt.
    \item Internal APIs must require valid JWT headers.
    \item Sensitive documents must be stored with restricted access.
\end{itemize}

\subsection{Performance}
\begin{itemize}
    \item Average API response time shall be under 200ms.
    \item Frontend initial load time shall be under 2 seconds on standard broadband.
\end{itemize}
